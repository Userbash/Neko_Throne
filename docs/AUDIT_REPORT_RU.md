## 2. Классификация найденных проблем

### 2.1. Критические

#### C-01: Использование объекта после deleteLater() — HTTPRequestHelper

**Модуль:** `src/global/HTTPRequestHelper.cpp`  
**Тип:** Use-after-free (потенциальный)  
**Серьёзность:** Критическая

**Описание:**  
В методе `NetworkRequestHelper::DownloadAsset` вызов `_reply->deleteLater()` происходит на строке 161, после чего на строках 162–174 происходит обращение к уже помеченному на удаление объекту: `_reply->error()`, `_reply->errorString()`, `_reply->readAll()`.

`deleteLater()` откладывает удаление до следующей итерации event loop. Пока нет повторного входа в event loop, доступ технически безопасен, но это нарушает принцип корректного владения объектом и создаёт риск некорректного поведения при будущих изменениях.

**Дополнительная проблема:** `_reply->readAll()` после `_reply->deleteLater()` читает уже буферизованные данные, но в случае повторного входа в event loop данные могут быть частично или полностью освобождены.

**Воспроизведение:** Наблюдаемо при каждом вызове `DownloadAsset`.

**Исправление:** Читать все данные в локальный буфер ДО вызова `deleteLater()`.  
**Статус:** ✅ Исправлено в данном PR.

#### C-02: Некорректная обработка пути в DownloadAsset — CoreManager

**Модуль:** `src/global/HTTPRequestHelper.cpp`, `src/sys/CoreManager.cpp`  
**Тип:** Логическая ошибка / некорректный путь к файлу  
**Серьёзность:** Критическая

**Описание:**  
Метод `DownloadAsset(url, fileName)` всегда строит путь как `Configs::GetBasePath() + "/" + fileName`. Когда `CoreManager::downloadLatestCore` передаёт в качестве `fileName` абсолютный путь (например, `/home/user/.config/Throne/core/NekoCore.update`), итоговый путь будет некорректным:  
`/home/user/.config/Throne//home/user/.config/Throne/core/NekoCore.update`

Это приводит к тому, что файл сохраняется не туда, куда ожидает `CoreManager`, и последующие вызовы `QFile::remove(tempPath)` и `QFile::rename(tempPath, destPath)` работают с неверными путями. Функция обновления ядра фактически сломана.

**Воспроизведение:** Попытка обновить ядро через меню приложения — бинарник не обновляется.

**Исправление:** Если `fileName` является абсолютным путём, использовать его напрямую без добавления базового пути.  
**Статус:** ✅ Исправлено в данном PR.

#### C-03: Гонка данных в CoreProcess::Restart()

**Модуль:** `src/sys/Process.cpp`  
**Тип:** Состояние гонки (race condition)  
**Серьёзность:** Высокая

**Описание:**  
Метод `Restart()` устанавливал `restarting = false` синхронно сразу после вызова `Start()`. Поскольку `QProcess::start()` асинхронен, при следующей итерации event loop мог сработать обработчик `stateChanged(NotRunning)` (от старого завершающегося процесса) с уже сброшенным флагом `restarting`. Это приводило к нежелательному повторному вызову `profile_stop(true, true)` в середине рестарта.

Дополнительно: `waitForFinished(1500)` может завершиться по таймауту без убийства процесса. После этого `Start()` вызывается на уже занятом `QProcess`, и новый процесс фактически не стартует.

**Воспроизведение:** Принудительное завершение ядра извне при запущенном профиле.

**Исправление:**  
1. После таймаута `waitForFinished` добавлен принудительный `kill()`.  
2. `restarting = false` сдвинут через `QTimer::singleShot(200, ...)` — это обеспечивает обработку всех ожидающих сигналов `stateChanged` до снятия флага.  
**Статус:** ✅ Исправлено в данном PR.

### 2.2. Высокая серьёзность

#### H-01: Отсутствие явного конечного автомата состояний ядра

**Модуль:** `src/sys/Process.cpp`, `include/sys/Process.hpp`  
**Тип:** Архитектурный  
**Серьёзность:** Высокая

**Описание:**  
Состояние жизненного цикла ядра отслеживалось через несвязанные булевы переменные:
- `Configs::dataStore->core_running` — запущено ли ядро
- `CoreProcess::started` — запускался ли процесс
- `CoreProcess::restarting` — выполняется ли рестарт
- `CoreProcess::failed_to_start` — провалился ли запуск

Эти флаги дублируют информацию и могут рассогласоваться. Нет единого источника истины о состоянии ядра.

**Исправление:** Добавлен enum `CoreLifecycleState`:
```cpp
enum class CoreLifecycleState {
    Stopped, Starting, Running, Stopping, Restarting, Failed
};
```
Каждый переход в `Process.cpp` теперь явно обновляет `m_state`.  
**Статус:** ✅ Исправлено в данном PR.

**Модуль:** `src/sys/Process.cpp`  
**Тип:** Производительность / UX  
**Серьёзность:** Высокая


**Статус:** ✅ Опечатка исправлена. Проблема с блокировкой главного потока частично смягчена (добавлен force-kill).

### 2.3. Средняя серьёзность

#### M-01: Утечка объекта GroupUpdater

**Модуль:** `src/configs/sub/GroupUpdater.cpp`  
**Тип:** Утечка памяти  
**Серьёзность:** Средняя

**Описание:**  
```cpp
GroupUpdater *groupUpdater = new GroupUpdater;
```
Глобальный голый `new` без владельца. Объект никогда не удаляется явно. В Qt при наличии родителя (`QObject` иерархия) или использовании `QApplication` это не приводит к видимой утечке при нормальном завершении, но нарушает принципы управления ресурсами.

**Статус:** 🔲 Не исправлено (низкий практический риск, требует более широкого рефакторинга).

---

#### M-02: QNetworkAccessManager создаётся на стеке в HttpGet/DownloadAsset

**Модуль:** `src/global/HTTPRequestHelper.cpp`  
**Тип:** Производительность  
**Серьёзность:** Средняя

**Описание:**  
`QNetworkAccessManager` создаётся локально при каждом запросе. Согласно документации Qt, рекомендуется переиспользовать один экземпляр `QNAM` для группы запросов (чтобы переиспользовать соединения и кэш). Создание нового экземпляра при каждом вызове неэффективно.

**Статус:** 🔲 Не исправлено (требует архитектурного изменения — выделения QNAM в синглтон).

#### M-03: Отсутствует таймаут в HttpGet для ситуации зависания

**Модуль:** `src/global/HTTPRequestHelper.cpp`  
**Тип:** Надёжность сети  
**Серьёзность:** Средняя

**Описание:**  
`accessManager.setTransferTimeout(10000)` установлен в `HttpGet`, но в `DownloadAsset` таймаут отсутствует. Загрузка большого файла (ядра или обновления) может зависнуть без таймаута.

**Статус:** 🔲 Не исправлено.

#### M-04: Режим блокировки не восстанавливается после перезапуска

**Модуль:** `src/sys/ProxyStateManager.cpp`  
**Тип:** Корректность  
**Серьёзность:** Средняя

**Описание:**  
При включённом Kill Switch и рестарте приложения состояние блокирующих маршрутов не восстанавливается. Это может привести к ситуации, когда пользователь думает, что Kill Switch активен, а трафик на самом деле проходит.

**Статус:** 🔲 Не исправлено.

### 2.4. Низкая серьёзность

#### L-01: Устаревший флаг restarting — не атомарный

**Модуль:** `include/sys/Process.hpp`  
**Тип:** Потокобезопасность  
**Серьёзность:** Низкая

**Описание:**  
`bool restarting` — обычный `bool`, не `std::atomic<bool>`. Если сигналы обрабатываются из вторичного потока (через `Qt::QueuedConnection`), возможна гонка на уровне кэша процессора.

В текущей архитектуре сигналы обрабатываются в главном потоке, так что практический риск минимален.

**Статус:** ⚠️ Учтено (флаг `m_state` типа `CoreLifecycleState` сейчас также не атомарный; при переносе рестарта в поток необходимо использовать `std::atomic`).
